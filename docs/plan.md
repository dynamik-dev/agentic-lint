# agentic-lint improvement plan

A review of the repo against what Anthropic/Cursor would ship, plus the token-efficiency moves that don't expand the config surface. Ranked, with file/line references. Read end-to-end once, then pick what to execute.

---

## 1. Top-line assessment

The architecture is right: two-phase script-then-semantic evaluation, glob-scoped rules, opt-in telemetry, a dedicated evaluator subagent with a strict output contract. The bones would pass an Anthropic design review.

What holds it back is the space between a clever prototype and a 1.0 product. Specifically: a hand-rolled YAML dialect that fails silently, a single example rule pack, no baseline mechanism for existing codebases, verbose skill prompts that tax every edit, and no LICENSE / CHANGELOG / SECURITY.md. Fix those, trim the skill prompts, and you have a tool people will actually adopt instead of admire.

---

## 2. The single biggest weakness

**The config format is a fragility bomb.** `pipeline.py:101` (`parse_config`) is a hand-rolled pseudo-YAML reader that handles a subset of YAML: 2/4-space indents, folded `>` scalars, inline `[a, b, c]` lists, hand-rolled comment stripping. Missing: `|` literal blocks, anchors, nested mappings, multi-line lists, `null`/bool coercion, any error messages whatsoever.

The README's pitch is "one config file." The config is actually a dialect. Malformed rules are silently dropped. A user who misindents a `script:` field gets a passing hook and a rule that never runs, with zero feedback.

Every shipped lint tool parses real config. The `stdlib-only` invariant is principled but traded a `pip install pyyaml` (7 MB) for a real chance that a rule quietly doesn't load. This is the first thing a maintainer at Anthropic would call out, and the first thing a would-be adopter will hit.

**Fix it first.** Everything else in this document is easier on a foundation that can validate its input.

---

## 3. Structural improvements (ranked by impact × ease)

### 3.1 Parse real YAML, schema-validate, fail loud
**Why:** fragility is the biggest adoption risk. Silent drops are worse than a dependency.
**What:** drop `parse_config` in `pipeline/pipeline.py`. Use `yaml.safe_load` (vendor PyYAML as a single file if stdlib-only is a hill to die on). Add a JSON Schema validator that rejects unknown keys, wrong types, invalid severity/engine enums, malformed globs — with line-number-pointing errors. Add `python3 -m agentic_lint validate` and call it once per session from `hook.sh`, not per edit.

### 3.2 Ship a starter rule library with `extends:`
**Why:** today `examples/laravel-package.yml` is the only pack. Every ESLint/Rubocop ecosystem win came from plugins. A user on Next.js or Django has nothing. Rule reuse separates a product from a template.
**What:** create `examples/packs/{react-ts,nextjs,django,fastapi,go,rails,rust-cli}.yml`. Add `extends: ["@agentic-lint/react-ts"]` to the schema; resolve at parse time.

### 3.3 Add a baseline file and per-line disables
**Why:** adopting lint on a codebase with 200 pre-existing violations is the #1 real-world blocker. Nobody does a 200-file cleanup day one. Ruff, ESLint, PHPStan all ship baselines.
**What:** add `.agentic-lint/baseline.json` with `{rule_id, file, line, checksum}` entries generated by `agentic-lint baseline init`. In `run_pipeline` (pipeline.py:485), filter violations against the baseline before blocking. Parse `# agentic-lint-disable: rule-id reason` comments per-line.
**Note:** `agentic-lint-init/SKILL.md:54` explicitly rejects baselines as "suppression lists, not rules." That's a value judgment your users won't share. Revisit.

### 3.4 Close the semantic-verdict telemetry loop
**Why:** `docs/telemetry.md:82,164` admits the gap. Semantic rules log `evaluate_requested` but never the verdict. `analyzer.py`'s noisy/dead classification is script-only. The review skill's value proposition assumes verdicts exist. Today a semantic rule the LLM never flags is indistinguishable from one fired constantly.
**What:** have the `agentic-lint` skill post a verdict record after dispatching the evaluator. Simplest: a `Stop`-triggered secondary hook (or write directly) that calls `python3 pipeline/log_verdict.py --rule X --verdict pass|violation`. Without this, "self-improving" is marketing.

### 3.5 Fix the synthetic-line-number footgun
**Why:** `pipeline.py:230–242` falls back to a diff with wrong line numbers when anchoring fails. The `agentic-lint` skill (SKILL.md:96) tells the agent "don't fabricate line numbers" — but your pipeline just did. The evaluator will cite those wrong numbers.
**What:** when the fallback triggers, emit a diff without line anchors and include `warning: "could not anchor to file on disk, line numbers are synthetic"` in the payload. Log this case in telemetry as a new record.

### 3.6 Kill the `jq` runtime dependency
**Why:** every prerequisite is friction. `jq` exists only to pull three fields out of JSON in `hook.sh`. The pipeline already parses stdin (pipeline.py:607).
**What:** `hook.sh` becomes `cat | python3 pipeline.py --hook-mode`. Drop the `STDOUT_FILE` / `STDERR_FILE` mktemp dance (hook.sh:53–55). Remove `jq` from README prereqs.

### 3.7 Add optional `fix_hint` on script rules
**Why:** docs argue against a `fix` field (design.md:72) but autofix is the single largest productivity delta ESLint delivers over prose-only linters. The evaluator already returns `fix:` lines; formalize the channel for script rules too. Cuts re-edit loops roughly in half.
**What:** optional `fix_hint: "replace with ..."` on script rules. Pipeline passes it as `suggestion` on the `Violation`. Skill already consumes `suggestion`.

### 3.8 Trim the skill prompts by 60%
**Why:** SKILL.md is injected into the model on every matching trigger. `skills/agentic-lint/SKILL.md` is 174 lines; `agentic-lint-author/SKILL.md` is 422 lines (a manual, not a prompt). Anthropic's `verification-before-completion` is ~40 lines.
**What:** keep the two exit-code formats, the dispatch instructions, the severity table. Cut "Troubleshooting" (speculative), "Performance Notes" (content-free), self-evident examples. Reorder: purpose → happy-path example → format reference → edge cases.

### 3.9 Strengthen the test suite
**Why:** 66 tests, but many are single-assertion. No property tests on `parse_config` with malformed input. No fuzz on `parse_script_output`'s regex cascade (pipeline.py:300–380). No subprocess-level contract test for `hook.sh`. `test_pipeline.py:50` asserts status without checking *why* — a bug that mislabeled every status as "pass" would slip through.
**What:** add Hypothesis property tests for the parser. Golden-file test for `_format_blocked_stderr`. Subprocess test that pipes synthetic `tool_input` JSON into `hook.sh` and asserts valid `hookSpecificOutput` JSON. Run `test-quality-reviewer` on the suite itself.

### 3.10 Rewrite the README landing page
**Why:** new users read 27 lines of YAML before seeing the loop in action. Install is 6 steps before "make an edit." The "Why" paragraph (line 61) reads as defensive rather than promissory.
**What:** open with a 5-line transcript — agent edits file → hook blocks → agent fixes → pass. Asciinema or plain text. Move install below Quick Start. Collapse four symlink commands to a one-line installer (`curl -sSL .../install.sh | bash`). Delete the bottom "Layout" tree (line 245) — nobody reads repo trees in READMEs.

## 4. Token efficiency (ranked, all zero-config)

Every item below is pipeline-side or skill-side. The `.agentic-lint.yml` surface doesn't change. No new fields, no new babysitting.

### 4.1 Trim SKILL.md files
Largest lever because it's per-edit, not per-eval. Cutting `agentic-lint/SKILL.md` from 174 → ~60 lines saves roughly 400 tokens **every matching edit**. `agentic-lint-author/SKILL.md` at 422 lines is worse; most of it is manual-style content.

### 4.2 Pipeline-side "can't possibly match" filters
Before dispatching the evaluator, skip semantic rules on diffs that can't violate them:
- Whitespace-only hunks → skip all rules.
- Comment-only hunks → skip rules that target identifiers/structure.
- Pure-deletion hunks → skip rules that only care about additions.
- Fewer than 2 added lines → skip semantic entirely.

Pure diff-content heuristics in `pipeline.py`. Invisible to the user. Saves the whole subagent dispatch on 30–50% of edits.

### 4.3 Verdict cache keyed by `(rule_id, hunk_sha256)`
Store `.agentic-lint/verdict_cache.json`, opt-in via directory presence (same pattern as telemetry). Identical hunks re-evaluated in a session skip the subagent. Huge on iterative fix-refire-fix loops. Zero config.

### 4.4 Drop `passed_checks` from the evaluator payload
`pipeline.py:505–509` sends it to the subagent. The subagent doesn't use it for judgment — only the parent agent consumes it, and the parent already has it. Passing it to the subagent is ceremony.

### 4.5 Inline eval below a threshold
If the diff is <15 lines and ≤1 semantic rule pending, skip the subagent; parent agent evaluates inline. Subagent boot overhead is system prompt + tool registry + agent definition (~60 lines + tool list). Trade: +60 lines parent context vs. ~2000 tokens subagent boot. Net save for small diffs.

### 4.6 Short-circuit auto-generated files
Pipeline-level hardcoded skip list for lockfiles and build artifacts:
```
package-lock.json, yarn.lock, pnpm-lock.yaml, poetry.lock, Cargo.lock,
*.min.*, dist/**, build/**, __pycache__/**, *.generated.*
```
These never benefit from lint but fire the hook and burn parse + scope-match cycles. No config — just an allowlist at the top of `pipeline.py` or in `hook.sh`.

### 4.7 Cap Write-mode file content
`build_diff_context` on Write sends the whole file via `_line_number(current)` (pipeline.py:222). For large files this blows up fast. Cap at N lines; if larger, slice to first+last K lines around the modified region (git-aided if available, first 200 lines with `truncated: true` marker if not).

### 4.8 Tighten the evaluator's system prompt
`agents/agentic-lint-evaluator.md` is 62 lines. Most is explanatory prose the evaluator doesn't need — the input JSON self-describes. Cut to ~20 lines: "Given the JSON below, return VIOLATIONS/NO_VIOLATIONS in this format." The parser in the parent skill enforces the contract; the evaluator just follows it.

### 4.9 Enable Anthropic prompt caching on the evaluator invocation
The evaluator's system prompt + rule definitions are stable across dispatches in a session. If Claude Code's subagent dispatch exposes `cache_control` (verify via Context7), mark the static block. Potential **80% reduction** in eval-side input tokens after the first call.

### 4.10 Force one-line evaluator verdicts
Evaluator output contract already has a shape; the **reasoning prose per rule** is the bloat. Tighten the system prompt to "one line per verdict, no reasoning prose." Saves output tokens on every dispatch.

## 5. The 80/20

If you only do three things:

1. **#3.1 (real YAML + schema + validate)** → makes the tool feel 1.0.
2. **#3.2 (starter packs + `extends:`)** → makes it useful day one, not day thirty.
3. **#3.4 (semantic verdict telemetry)** → makes the self-improvement loop real.

If you only do three things for tokens:

1. **#4.1 (trim skill prompts)** → per-edit savings compound immediately.
2. **#4.2 (can't-match filters)** → avoids entire subagent dispatches.
3. **#4.3 (verdict cache)** → near-free payoff on iterative loops.

The combined six-item list moves per-edit token cost roughly in half while making the tool feel shipped instead of hacked.

## 6. What's already strong (don't break)

- **Two-phase script-then-semantic engine.** `passed_checks` feeding into the evaluator context is a sharp touch.
- **Evaluator output contract.** `VIOLATIONS:` / `NO_VIOLATIONS:` shape in `agents/agentic-lint-evaluator.md:37–48` is clean and parser-friendly.
- **Opt-in telemetry by directory presence.** `_telemetry_path` in `pipeline.py:449–460` is exactly right — no surprise files, no flag-in-config.
- **Dogfooding.** `.agentic-lint.yml` linting the pipeline itself is credibility. The `stdlib-only` rule is a particularly good use of the tool.

## 7. Anthropic/Cursor polish specifics

### 7.1 Tighten skill `description:` frontmatter
Yours (`skills/agentic-lint/SKILL.md:3`) is 4 sentences, ~400 chars. Anthropic's convention is 1 sentence, ~150 chars. This is what the router reads when deciding to activate. Shorten.

### 7.2 Reorder SKILL.md bodies
Yours lead with format enumerations. Anthropic's lead with the canonical happy-path example. New ordering: one-sentence purpose → happy-path example → format reference → troubleshooting (if any).

### 7.3 Version the schema
Add `schema_version: 1` at the top of `.agentic-lint.yml`. Every shipped config format has it — ESLint, Biome, Cursor's `.cursorrules`, Anthropic's skill frontmatter. Without it, breaking changes silently corrupt old configs.

### 7.4 Ship `agentic-lint doctor`
Every polished tool has a diagnostic. README step 6 is `--help`, which proves Python runs — it doesn't prove the hook is wired, the evaluator agent is registered, or the config parses. A `doctor` command checking all five install steps replaces your first support ticket.

### 7.5 Missing repo hygiene files
No `LICENSE`, `CHANGELOG`, `CODE_OF_CONDUCT.md`, `SECURITY.md`. No serious adopter merges into a repo without a LICENSE. Add all four before promoting the project.

### 7.6 Fix scope/description mismatches in the dogfood config
Caught while writing this very document. `.agentic-lint.yml:38–45` defines `no-orchestration-labels` with `scope: "*"` (matches every file) but the description says "Source code must not carry project-management artifacts." That mismatch fired this doc into the evaluator for a rule the rule itself scopes to source code. A capable evaluator handled it correctly; a cheaper one might not.

Two takeaways:
1. Tighten the scope on that specific rule to source globs (e.g. `["*.py", "*.ts", "*.tsx", "*.go", "*.rs", "*.php", "*.rb", "*.js", "*.sh"]`) so it doesn't fire on docs/configs/markdown.
2. More broadly, a linter-of-the-linter check is worth building: the `agentic-lint-author` skill or a new `validate` subcommand should warn when a rule's description claims to apply to a narrower set than its `scope` admits — scope drift is a real class of bug and it shows up immediately in wasted evaluator dispatches.

---

## 8. Execution order

A pragmatic sequence that keeps the tool usable at every step:

1. `LICENSE` + `schema_version: 1` — trivial, protects everything downstream.
2. Real YAML + schema validation (#3.1) — foundation for safe iteration.
3. Trim skill prompts (#3.8 + #4.1) — immediate per-edit token savings.
4. Starter rule packs + `extends:` (#3.2) — unlocks day-one adoption.
5. Verdict telemetry (#3.4) — makes the loop real.
6. Baseline mechanism (#3.3) — removes the main adoption blocker.
7. Token-efficiency batch (#4.2, #4.3, #4.6) — compound wins.
8. README rewrite (#3.10) — do last, after there's something to show.

Everything else — test rigor, `doctor`, autofix hints, synthetic-line-number fix, `jq` removal — can be interleaved opportunistically.
